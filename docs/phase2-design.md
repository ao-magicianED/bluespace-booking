## 2. 管理者ダッシュボード（/admin）

- 認証: `ADMIN_PASSWORD`（環境変数）＋署名Cookie。1人運用なのでこれで十分（フェーズ3で複数人ならSupabase Auth）
- 予約一覧: 状態フィルタ（今後/確定/未入金/キャンセル）、検索（名前・メール）
- 予約詳細: **キャンセル＆返金ワンクリック**（Stripe返金→カレンダー削除→DB更新→お詫びメールを1ボタンで実行。現在の手動4ステップを自動化）
- 異常検知パネル: calendar_sync_status=failed、検証エラーの予約を強調表示
- 請求書払いの入金待ち一覧＋「入金確認→確定」手動ボタン（自動化の保険）

## 3. 通知

- 確定時: 利用者にメール（Resend）/ 管理者に **メール＋Discord Webhook**
- Discord Webhookは1行設定（URL環境変数）で即時スマホ通知。既存のBOT運用（返信BOT等）と同じ場所に流せる
- フェーズ2で追加する通知: 請求書発行・入金確認・期限切れ自動キャンセル・前日リマインダー（利用者向け）

## 4. 会員制マイページ（2026-06-11 方針変更: ログイン制を採用）

- `/my`（要ログイン）: 予約履歴一覧・予約詳細・入退室案内・キャンセル申請

## 8. 請求書払いフロー（調査結果ふまえた設計）

### 調査結果（2026-06-11 freee API実地確認）

| 項目 | 結果 |
|---|---|
| freee会計API（入金明細・取引） | ✅ 接続OK。銀行口座連携済み（PayPay銀行・GMOあおぞら等がAPI連携中） |
| freee請求書API（/iv） | ⚠️ 現在のトークンでは**401（権限なし）**。freeeアプリ側で「freee請求書」のスコープを追加して再認証すれば使える見込み |
| freee請求書の**メール自動送付** | ❌ APIに送付エンドポイントが**ない**（発行まで。送付はfreee画面からワンクリック） |
| freeeの入金Webhook | ❌ ない → 入金検知は**ポーリング**（定期的にAPIを見に行く）になる |
| 振込と予約の自動突合 | ⚠️ 振込名義のゆれ問題あり（「カ）ブルーステージ」等）。金額＋名義のあいまい一致で誤検知リスク |

### 結論: 「freee完全自動」は難しい。現実的な選択肢は3つ

**案A: Stripe請求書＋銀行振込（全自動・推し）**
- Stripeの請求書機能で請求書メール自動送付。支払期限も設定可能
- 銀行振込を選ぶと**顧客ごとの専用振込口座**をStripeが発行 → 名義ゆれ問題が構造的に消える
- 入金されると即Webhook → **予約自動確定**（カード決済と同じ仕組みに乗る。実装も自然）
- 適格請求書: 登録番号を請求書に記載可能
- デメリット: 手数料（請求書機能0.4%＋銀行振込手数料。合計1〜2%程度）。freeeには入金データとして記帳される（既存のStripe売上と同じ流れ）

**案B: freee発行＋freee自動消込のポーリング（半自動）**
- API（権限追加後）で請求書を自動発行 → **送付だけfreee画面からワンクリック**（ここが手動）
- 入金はfreeeの口座連携＋自動消込にまかせ、システムが1時間ごとに消込状態をポーリング → 消込済みになったら予約自動確定
- 完全無料（振込手数料は顧客負担）。ただし「送付クリック」と「消込の精度」に人が絡む

**案C: freee発行＋入金確認だけ人間（堅実・最小実装）**
- 請求書はAPIで自動発行・管理画面に「入金確認→確定」ボタン
- 通帳 or freeeで入金を見たらボタンを押すだけ（1クリック）。確定処理・カレンダー登録・メールは全自動
- 月数件のうちは一番事故が少ない

### 期限ルール（ユーザー要望の整理）

- 請求書払いの支払期限: **発行から3日後**
- **利用開始まで3日（72時間）を切っている予約は請求書払いを選べない**（カードのみ）
- 利用開始が3〜6日先の場合: 期限を「3日後」ではなく**利用開始の24時間前**に短縮（比例ルール）
- 期限までに入金がなければ**自動キャンセル**（枠解放＋通知）。枠は期限まで仮押さえ扱い

### 状態設計の拡張

- `payment_method`: card / invoice を追加
- 請求書払いの予約は `booking_status='awaiting_payment'`（枠はブロック、排他制約の対象）
- 入金確認 → confirmed に遷移（カレンダー登録・通知はカード決済と同じ処理を再利用）

## 8.5 料金体系（2026-06-11 追加要件）

フェーズ3予定だった pricing_rules を**フェーズ2に前倒し**する。

### 基本料金（京成小岩・予定）
- 平日: ¥1,000/時間
- 休日: ¥2,000/時間（★論点: 「休日」の定義 = 土日のみ or 土日＋祝日。祝日を含むなら祝日判定データを持つ）

### 割引（リードタイム連動 = 動的価格の第一歩）
- **直前割**: 利用当日の予約は10%OFF
- **早割**: 利用日の30日以上前の予約は10%OFF
- ★論点: 割引の併用なし（排他）でよいか／割引はオプション料金には掛けない想定
- 実装: `discount_rules`（種類・条件日数・割引率・有効フラグ）をvenue単位で持ち、価格計算はすべて `pricing.ts` 1箇所に集約。breakdown に「基本料金・適用割引・オプション」を明記してメール・領収書にも内訳表示

### スキーマ
```
pricing_rules: id, venue_id, day_type('weekday'|'weekend'|'holiday'), price_per_hour, valid_from, valid_to
discount_rules: id, venue_id, kind('last_minute'|'early_bird'), days_threshold, percent_off, active
```
