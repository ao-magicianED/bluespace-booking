### 重要な発見

1. **スペイシーは2026年3月にJR西日本「+PLACE」と統合し「Yoyappin（ヨヤッピン）」になった**
   （約14,000拠点）。なお「予約便」という機能は存在しない。最も近いのは「まとめて予約」
   （同一スペースの複数日程を一括予約する機能）で、聞き間違いの可能性が高い。
2. **3社共通のCVR（予約率）の柱は「実利用者レビュー」**。星評価＋件数＋利用者属性（用途）＋
   ホスト返信のセット。インスタベースは「利用者の単体レビュー」と「掲載者の返信レビュー」を
   構造的に分離して自演を防止。スペースマーケットは相互ブラインド公開方式。
3. **スペースマーケット公式データ: 「今すぐ予約（即時確定）」設定ありは無しの平均2倍予約が入る**
   （成約の7割超が即時予約）。→ 本サイトは最初から全拠点即時確定なので優位。この強みを表示で訴求すべき。
4. **写真は15枚以上推奨**が業界標準（スペースマーケット最大30枚・推奨15枚以上16:9、
   インスタベース最低15枚、スペイシー10〜15枚以上）。1枚目（サムネイル）の質が予約率を左右。
5. **料金の柔軟性**: 3社ともパック料金（終日・時間パック）・時間帯/曜日別プラン・人数課金・
   オプション課金を提供。本サイトは「平日/土日祝×時間単価＋オプション」のみ。
6. **キャンセルポリシーの事前明示**が3社標準（スペースマーケットは易/普/厳/カスタムの4プリセット）。
7. その他の共通機能: お気に入り、ポイント還元（インスタ3%/180日・スペマはランク制最大10%）、
   用途タグ検索・用途別特集ページ（SEO流入源）、メッセージ機能、本人確認バッジ、保険自動付帯。

### 本サイトの既存優位（競合にない強み・変更不要）

- 仲介手数料なし（Stripe 3.6%のみ）＝常に最安値保証
- 全拠点が即時確定・開始1分前まで予約可・30分単位
- 直前割/早割/自動クーポン（初回・2回目・掘り起こし）/請求書払い/領収書/時間変更セルフ申請

## 4. レビューシステム設計（実装済み・2026-07-10）

### 4-1 フロー

```
利用終了（end_at経過）
  → cron（毎日JST3時）が利用終了後の確定予約を検出しレビュー依頼メール送信
     （bookings.review_request_sent_at で冪等・直近3日以内終了分のみ＝復旧時の大量送信防止）
  → お客様が /review/[review_token] から投稿（星1〜5・コメント・用途・表示名）
     ※会員はマイページ予約詳細の「レビューを書く」からも投稿可
  → booking_reviews に status='pending' で保存＋管理者にDiscord/メール通知
  → 管理者が /admin/reviews で承認（published）/ 非公開（rejected）/ 運営返信
  → 拠点ページに公開（平均★・件数・一覧・返信）＋ LocalBusinessのAggregateRating出力
```

### 4-2 設計判断（なぜこうしたか）

- **実予約者のみ投稿可**（review_token = 予約ごとの秘密UUID）: 競合3社と同じ「実利用者だけ」
  の信頼性を担保。1予約1レビュー（booking_idにUNIQUE制約が本命ガード）
- **承認制（デフォルト非公開）**: 無人運営のため誹謗中傷・個人情報の露出をワンクッションで防ぐ。
  インスタベースの「単体レビュー/返信レビュー分離」を簡略化した形
- **AggregateRatingは実レビューが1件以上のときだけ出力**: 偽装評価とみなされるリスク回避
- **静的「ご利用者の声」（venues.ts）はフォールバック**: 実レビューが1件以上集まった拠点は
  実レビュー表示に自動で切り替わる（ReviewSection.tsx）
- **投稿受付はライブラリ利用終了後30日以内**（REVIEW_WINDOW_DAYS）: インスタ7日・スペマ14日より
  緩め＝件数を貯める初期フェーズを優先

## 6. 今後のロードマップ（Sonnet 5への実装指示書・優先度順）

> 共通ルール: 新規migrationは0017から採番（§5-1参照・0004が2本ある事故に注意）。
> 価格計算に触るものは必ず pricing.ts / quote.ts の「表示=決済の同一計算」を維持し、
> vitestを追加すること。デプロイ前に `npm test` と `npm run build`。

### 6-1 パック料金（優先度★★・工数中）

- venues に `packs jsonb` を追加（例: `[{"hours":3,"price":2700,"label":"3時間パック"}]`）
- pricing.ts を v3 に拡張: 選択時間がパック時間と一致したら「時間単価×h」と比較して安い方を自動適用
  （インスタベース方式の「最安プラン自動選択」＝ゲストに損をさせない）
- price_breakdown に `pack` フィールドを追加（rule: "v3"）。**過去予約のv2は再現できるよう
  calcQuoteのv2ロジックは削除しない**こと
- BookingGrid の見積もり表示に「🉐 3時間パック適用で¥300お得」のような表示
- テスト: パック適用/非適用の境界・割引/クーポンとの併用順序

### 6-2 用途別ランディングページ（優先度★★・工数小〜中・SEO）

- `/uses/[use]/page.tsx` を新設（会議・パーティー・撮影・レッスン・ボードゲーム等8種目安）
- venues.ts の `uses` 配列に用途slugのマッピングを追加（`useTags: string[]`）し、
  該当拠点カード＋用途別の訴求コピー＋FAQを表示
- 拠点ページの用途チップ（uses-chips）から各用途ページへリンク
- sitemap.ts に用途ページを追加。ItemList構造化データ
- 注意: 「東京 会議室 30分単位」等のロングテールを狙う。titleは「用途×駅名」で組む

### 6-3 時間帯別料金（優先度★★・工数大・慎重に）

- venues に `time_band_prices jsonb`（例: 深夜0-6時は0.8倍、18-24時は1.2倍）
- slots.ts の DaySlots.pricePerHour が「日単位」前提のため、スロット単位の価格へ拡張が必要
  （availability API・BookingGrid の表示・quote.ts の検証まで影響。**5点照合を壊さないこと**）
- 跨ぎ予約（20時〜翌1時等）の按分計算をテストで固める

### 6-4 まとめて予約＝複数日程の一括予約（優先度★★・工数大）

- Yoyappinの「まとめて予約」相当。ユースケースは毎週のレッスン・定期会議（現在は問い合わせ→
  管理者手動見積もりで対応中。これをセルフ化して「定期10%OFF」を自動適用）
- 設計方針: bookings に `group_id uuid` を追加し、1決済（Stripe Checkout 1セッション）で
  複数予約行を作る。1枠でも埋まっていたら全体エラーにする（排他制約で自然に守られる）
- キャンセルは1枠単位（グループ一括キャンセルは管理者のみ）

### 6-5 お気に入り・再予約（優先度★・工数小）

- member_favorites テーブル（user_id, venue_id, UNIQUE）
- マイページに「もう一度予約する」（前回と同拠点・同時間帯をプリセットして拠点ページへ）

### 6-6 ポイント還元（優先度★・工数中）

- 既存の自動クーポン（初回/2回目/掘り起こし）と役割が重なるため、**導入前に費用対効果を検討**。
  やるなら「利用額の3%を次回クーポンとして自動発行」（couponsの仕組みを流用・restrict_email）
  が最小実装。新テーブル不要

### 6-7 人数追加料金（優先度★・設計のみ）

- venue_options に「追加人数（per_person）」price_unit を足す案と、venuesに
  `base_capacity`/`per_person_price` を足す案がある。後者は quote 5点照合の変更を伴うため、
  まずはオプション拡張（per_person × party_size）で始めるのが安全

### 6-8 月表示カレンダー（優先度★・工数中）

- 拠点ページに「月の空き概況」（○△×の3値・タップで該当週の予約グリッドへスクロール）
- /api/availability を28日分バッチ取得できるよう拡張（FreeBusyのquota注意・キャッシュ推奨）

### 実装しないと決めたもの（理由つき）

- **サイト内メッセージ/チャット**: 無人運営1人体制では返信SLAを守れず逆効果。問い合わせフォーム＋
  電話で十分。スペマの「返信率が検索順位に影響」は自社サイトでは無関係
- **検索・絞り込み**: 7拠点では一覧で足りる。用途別ページ（6-2）が実質の絞り込みになる
- **本人確認バッジ**: プラットフォームと違い運営者=掲載者なので不要。特商法ページ整備が代替
- **保険の自動付帯**: コードでなく保険契約の話。事業判断としてToDoに記載のみ

## 7. 掲載品質チェックリスト（競合の掲載マニュアル基準・コード変更不要の運用改善）

3社の掲載マニュアルから抽出した「ページ品質」基準。管理UI（/admin/venues）で対応可能:

- [ ] 写真15枚以上/拠点（現状: import-photos-v2.mjs で取り込み済み。不足拠点は追加撮影）
- [ ] 1枚目（hero）は「スペース全体が映る明るい写真」（0.2秒で第一印象が決まる）
- [ ] タイトルに「駅徒歩分数×収容人数×用途」を含める（venues.tsのcatchCopy見直し）
- [ ] 概要文にSEOキーワード（駅名・用途・設備）を自然に含める
- [ ] 設備リストの正確性（「あるのに書いてない」は検索露出の損）
- [ ] 利用ルール・注意事項の明文化（FAQでカバー済みか各拠点確認）
