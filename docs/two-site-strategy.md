# 2サイト連携戦略（コーポレート × 予約サイト）

作成: 2026-06-11 / ステータス: **確定**（Codexセカンドオピニオン反映済み）

## Codexレビュー結論（2026-06-11）

> 「この戦略で進めてよい。ただし bluespacerental.com を**予約も詳細も集約する唯一の正本**と決め切ること。
> コーポレートは会社信頼と事業概要に絞り、拠点SEO・GBP・予約CVは予約サイトへ集中させる」

反映した修正:
1. 表現修正: 「分離がSEO最強」ではなく「予約CV・実装・ブランド運用込みで分離が妥当」
   （純SEOならサブディレクトリ統合が最強だが、技術都合と予約CVを含めると分離が総合最適。
   コーポレートが新規ドメイン同然なので失うオーソリティも無い）
2. **コーポレートに拠点詳細を残し続ける設計は不可** → 完成後に必ず301で一本化する
3. **「早すぎる301」が最大リスク** → 予約サイト側の拠点ページが完成してから1対1で301。
   多段リダイレクト（.pages.dev→lcc.com→rental.com）は避け旧URL→最終URL直行。1年以上維持
4. インスタベース等は「敵」ではなく**発見・レビュー獲得チャネル**として併用継続。
   外部リンクは自拠点ページ直リンクのみ。GA4/UTMでチャネル別予約CVを計測
5. LocalBusinessスキーマ: @id/name/url/image/address/geo/telephone/openingHours/priceRange/
   parentOrganization/sameAs を実装。**名称・住所はGBPと完全一致**。レビュー構造化データは優先度低
6. 用途別LP量産・ブログ量産は**過剰施策**として却下。実データを見て需要があるLPだけ後から

## 現状整理

| | コーポレートサイト | 予約サイト |
|---|---|---|
| URL | bluestage-corporate.pages.dev（→将来 bluestage-lcc.com 予定） | **bluespacerental.com**（本番稼働中） |
| 技術 | Astro / Cloudflare Pages | Next.js / Vercel |
| 内容 | 会社案内・AI/エネルギー/民泊/不動産・**レンスペ7拠点詳細**（京成小岩=完成形） | 予約エンジン（決済・会員・領収書）。拠点1（京成小岩）、詳細コンテンツは薄い |
| 予約導線 | **CTAがインスタベース/スペマのトップページ行き**（自拠点ページですらない） | 自社予約完結（手数料ゼロ） |

### 現状の問題点（ユーザー視点）

1. **コーポレートで拠点に惚れた人を、競合市場のトップページに放流している**
   → インスタベースに着いた瞬間「他の安いスペース」が並び、戻ってこない。
   手数料30〜35%も取られる。最悪の導線。
2. 予約サイトに着地した人は逆に「写真・設備・アクセス」が薄く、判断材料が足りない。
3. 2サイトに同じ拠点詳細を置くと重複コンテンツでSEO評価が割れる。

## 推奨: 役割分担型（2ブランド2サイト）

```
【ブルースペース】bluespacerental.com = 予約・拠点ブランドサイト（刈り取り）
  - 7拠点の詳細ページ（写真・設備・アクセス・FAQ・空き状況・即予約）
  - ターゲットKW: 「レンタルスペース 上野」「貸し会議室 京成小岩」「パーティールーム 西新宿」
    …地名×用途の取引型(今すぐ借りたい)キーワード全部
  - LocalBusinessスキーマ×7拠点、Googleビジネスプロフィールのリンク先

【ブルーステージ合同会社】bluestage-lcc.com = コーポレート（信頼・指名）
  - 会社概要・事業一覧・採用・ニュース・民泊/AI/エネルギー事業
  - レンスペは「事業紹介1ページ」のみ（7拠点の一覧カード→予約サイトへリンク）
  - ターゲットKW: 「ブルーステージ合同会社」指名検索、B2B(法人・パートナー・外販)
```

### なぜ統合(1サイト)より分離か

- **検索意図が違う**: 「上野 レンタルスペース」で探す人は会社情報に興味がない。
  用途特化サイトの方がトピックの専門性が明確でSEOが強い（テーマが薄まらない）。
- **技術が違う**: 予約エンジンはNext.js/Vercel必須。コーポレートはAstro/CF Pages。
  無理に寄せるとどちらかを作り直しになる。
- **ブランドが立つ**: 「ブルースペース」で拠点ブランドを育てられる
  （インスタベース内の店舗名とも一致させられる）。
- ドメインは取得済み・接続済み（bluespacerental.com）。

## ユーザー導線（あるべき形）

```
発見経路A: Google検索「上野 レンタルスペース」
  → bluespacerental.com/ueno-4a（写真・料金・空き枠が1画面）
  → そのまま3クリックで予約 ✅

発見経路B: Googleマップ（ビジネスプロフィール）
  → ウェブサイトリンク = bluespacerental.com/該当拠点 → 即予約 ✅

発見経路C: インスタベース内で発見
  → 店舗名でGoogle検索 → 公式(bluespacerental.com)が出る
  → 「公式サイト予約が最安」で自社に転換 ✅（手数料35%→3.6%）

発見経路D: 会社を調べに来た（法人・長期・パートナー）
  → bluestage-lcc.com/services/rental-space（事業概要）
  → 拠点カード → bluespacerental.com/拠点 or 長期問い合わせフォーム ✅
```

### CTA設計（両サイト共通ルール）

- 第1ボタン: **「公式サイトで予約（最安・手数料なし）」→ bluespacerental.com/拠点**
- 第2ボタン: 長期・法人の相談（フォーム）
- 外部サイト（インスタベース等）リンクは**残すが格下げ**（テキストリンク・自拠点の掲載ページ直リンクに修正。トップページ行きは廃止）

## SEO実装計画

### Phase A: 予約サイトのSEO基盤＋拠点詳細の充実（最優先）
- [ ] metadata（title/description）拠点ごと、OGP画像
- [ ] sitemap.xml / robots.txt / favicon
- [ ] JSON-LD: LocalBusiness（住所・営業時間・地理座標）×拠点、BreadcrumbList
- [ ] 拠点ページにコーポレートの完成形コンテンツを移植
  （写真ギャラリー・設備・アクセス・利用シーン・FAQ・レビュー）
  → venuesテーブル拡張 or コンテンツファイルでデータ駆動化
- [ ] フッターに「運営会社: ブルーステージ合同会社」→コーポレートへリンク（E-E-A-T）

### Phase B: 残り6拠点を予約サイトに追加
- カレンダーIDと住所は判明済み（コーポレートCLAUDE.mdに7拠点分）
- 必要: 各拠点の料金（平日/休日）・写真・紹介文（コーポレートから移植）

### Phase C: コーポレート側の導線修正（すぐできる・効果大）
- [ ] 京成小岩ページのCTA: 第1=自社予約（bluespacerental.com/keisei-koiwa）に差し替え
- [ ] 他6拠点も追加され次第、順次差し替え
- [ ] 外部リンクをトップページ→自拠点の掲載ページ直リンクに修正

### Phase D: 重複解消（コンテンツ移植完了後）
- コーポレートの拠点詳細7ページ → **301リダイレクト**で予約サイトの対応ページへ
  （リンク資産を予約サイトに集約。コーポレートには事業概要1ページを残す）
- ※「302やcanonicalで様子見」でなく301が正解（恒久移転のため）

### Phase E: ローカルSEO・継続施策
- Googleビジネスプロフィール7拠点の「ウェブサイト」→ 予約サイトの各拠点ページに変更
- 拠点ページに「近隣情報」「使い方ガイド」等の独自コンテンツ追記
- 予約サイトに用途別LP（会議/撮影/パーティー×エリア）を順次追加

### 前提条件（重要）
- コーポレートサイトを **bluestage-lcc.com 本ドメインに載せる**（現在.pages.dev）。
  .pages.devのままではコーポレート側のSEOは育たない。
  ※ルートは現在Jimdo。移管PJの完成・切替が前提。

## 効果の見立て

- 導線修正（Phase C）だけで: コーポレート経由の予約が手数料35%→3.6%に
- Phase A+B完了で: 地名×用途KWでの流入が予約サイトに直接着地→最短導線で転換
- Phase D完了で: 「店舗名」検索で公式が1位に立ちやすくなる（重複が消え評価集約）
